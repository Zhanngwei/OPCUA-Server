# OPC UA 协议与项目源码学习指南设计说明

## 1. 目标

面向不了解 OPC UA 协议、但具备基本 Java/Web 开发经验的读者，制作一套“协议与源码并重”的学习指南。读者完成阅读和实验后，应能够：

- 解释 OPC UA 的信息模型、通信服务、订阅机制与安全模型。
- 将 Node、NodeId、Namespace、Reference、Endpoint、Session、Subscription 等概念对应到本项目源码。
- 启动服务器，使用 Web UI、REST API 和 UaExpert 完成建点、浏览、读写和订阅。
- 理解项目的启动流程、模块边界、核心数据流和持久化机制。
- 判断哪些能力来自 OPC UA 标准、哪些由 Eclipse Milo 提供、哪些由本项目实现。
- 按照现有扩展点新增业务节点，或将模拟值替换为真实设备数据。

## 2. 交付物

交付物统一存放在 `OPCUA-Source-Guide`：

- `README.md`：阅读入口、文件说明和推荐学习路径。
- `OPCUA源码学习指南.md`：适合源码仓库阅读的 Markdown 完整版。
- `OPCUA源码学习指南.html`：无需服务端、双击即可阅读的离线 HTML 完整版。

设计说明与实施计划分别放在 `OPCUA-Source-Guide/specs` 和 `OPCUA-Source-Guide/plans`。

## 3. 内容组织

指南采用单篇渐进式结构，按照“建立概念—映射源码—执行实验—理解边界—尝试扩展”的顺序展开：

1. 阅读地图与最短上手路径。
2. OPC UA 的产生背景、解决的问题和核心术语。
3. 信息模型：节点、属性、引用、地址空间、命名空间和 NodeId。
4. 通信模型：发现、端点、安全通道、会话、浏览、读写、订阅和监控项。
5. 安全模型：应用实例证书、消息安全、用户身份令牌和信任流程。
6. 项目总体架构、技术栈、目录结构和核心功能矩阵。
7. 启动链路与关键类导读。
8. 动态地址空间、节点 CRUD、NodeId、数据类型、访问级别和持久化。
9. Web UI、REST API、服务层、Milo 节点模型和 H2 之间的数据流。
10. UaExpert、Web UI 和 REST API 三组实战。
11. 二次开发：新增节点、接入真实数据源和扩展协议能力。
12. 调试、常见问题、安全加固与生产化检查清单。
13. OPC UA 标准能力与项目现状对照表。
14. 源码阅读索引与术语速查表。

## 4. 源码解释原则

- 每个关键协议概念至少关联一个本项目类、方法、配置项或实验步骤。
- 结论优先引用实际源码路径和行号，不以 README 描述代替源码事实。
- 明确区分三层责任：OPC UA 标准、Eclipse Milo SDK、本项目业务封装。
- 对未实现或仅部分实现的能力明确标注，不将 SDK 能力表述为项目已完成能力。
- 对代码与配置文件之间可能存在的偏差单独提示，例如静态端点配置与运行时代码行为。
- 示例命令以 Windows PowerShell 和当前仓库默认端口为准，同时保持核心命令跨平台可理解。

## 5. 图示设计

Markdown 使用 Mermaid 表达关系和时序；HTML 内嵌等价 SVG 或 HTML/CSS 图形，不依赖网络资源。至少包含：

- OPC UA 交互与安全分层图。
- 项目总体架构图。
- Spring Boot 与 Milo 启动时序图。
- 动态节点创建和 H2 持久化数据流图。
- 证书首次连接、拒绝、信任和重连流程图。
- Browse、Read/Write、Subscription 数据交互时序图。
- 核心类关系图。

图示颜色和标签在 HTML 深浅背景下均应可辨识；所有图均配文字说明，避免图形成为唯一信息来源。

## 6. HTML 设计

HTML 为单文件离线文档，采用内嵌 CSS 和少量原生 JavaScript：

- 左侧固定目录，正文为响应式阅读区。
- 小屏幕下目录可折叠。
- 支持章节锚点、返回顶部和当前章节提示。
- 使用卡片、表格、提示框和代码块区分协议、源码、实战与风险信息。
- 不加载 CDN、外部字体或远程脚本，避免离线打开时内容缺失。
- 图示使用内嵌 SVG/HTML，不依赖 Mermaid 运行时。

## 7. 实战范围

实战以本仓库现有能力为准：

- 使用 Maven 启动 Spring Boot 与 OPC UA Server。
- 登录 Web UI 并识别节点、端点、证书、用户和状态模块。
- 使用 UaExpert 连接 `opc.tcp://localhost:4840/milo`。
- 浏览 `Objects/HelloWorld`，读取和订阅动态变量。
- 通过 Web UI 创建文件夹和变量并修改变量值。
- 通过 REST API 登录、建点、查询地址空间和更新变量。
- 验证 H2 持久化节点在重启后的恢复过程。
- 演示客户端证书首次连接后进入拒绝状态、人工信任后重新连接的流程。

若某项操作依赖外部工具或当前运行环境，不把未执行的观察结果写成已验证事实，而是提供预期现象与排错路径。

## 8. 项目边界与风险提示

指南将建立能力矩阵，至少区分：

- 已有实现：地址空间浏览、动态文件夹和变量、读写、数据项订阅、节点持久化、端点展示、用户认证、证书状态管理、Web 管理界面。
- 部分实现或需核验：端点配置应用范围、连接统计精度、证书文件目录与运行时 PKI 的一致性。
- 未见完整业务实现：自定义事件模型、方法节点调用、历史数据访问、报警与条件、PubSub、LDS 注册和高可用部署。

安全章节必须提示默认账号、H2 凭据、无安全端点、用户名令牌策略、开发期 H2 Console 和证书目录等生产风险。

## 9. 验证标准

- 三个主要交付文件均存在，UTF-8 编码可正确显示中文。
- Markdown 标题层级、内部链接、代码围栏和 Mermaid 块成对完整。
- HTML 可被解析，目录锚点都有目标，CSS/JavaScript/SVG 均内嵌且无外部网络依赖。
- 所有引用的源码路径真实存在；抽样核对行号和代码结论。
- 默认端口、端点、启动命令和配置值与仓库当前源码一致。
- 标准能力与项目能力清晰分离，无 `TODO`、`TBD` 或空占位内容。

## 10. 非目标

- 不改动服务器、前端或数据库业务代码。
- 不声称覆盖全部 OPC UA 规范或替代 OPC Foundation 官方规范。
- 不新增需要联网加载的文档框架。
- 不把指南扩展为 API 自动生成系统或项目官网。
